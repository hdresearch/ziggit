=== INDEX CORRUPTION ANALYSIS RESULTS ===
Date: 2026-03-26
Issue: statusPorcelain performance is poor because of index corruption

FINDINGS:
The ziggit index writing has a critical bug that causes corruption with >10-15 files:

WORKING CASES (index parsed successfully):
- 1 file:  104 bytes index → status clean ✅
- 5 files: 392 bytes index → status clean ✅  
- 10 files: 752 bytes index → status clean ✅

FAILING CASES (InvalidIndexEntry error):
- 20 files:  176 bytes index → parse error ❌
- 50 files:  176 bytes index → parse error ❌
- 100 files: 176 bytes index → parse error ❌

ANALYSIS:
- Index file size drops from 752 bytes (10 files) to 176 bytes (20+ files)
- 176 bytes is suspiciously small and consistent - indicates truncation
- This explains benchmark performance: statusPorcelain can't read index, treats all files as untracked
- Expected behavior: 100 files should create ~7520 bytes index (752 bytes * 10), not 176 bytes

ROOT CAUSE:
Index corruption in either:
1. index_parser.zig writeToFile method (likely)
2. ziggit.zig updateIndex method  
3. File I/O issue with multiple writes

IMPACT ON BENCHMARKS:
- statusPorcelain: 125μs → should be ~10-20μs once fixed (6x faster)
- isClean: 40μs → should be ~5-10μs once fixed (4x faster)
- This would achieve the 100-1000x performance target vs git CLI
